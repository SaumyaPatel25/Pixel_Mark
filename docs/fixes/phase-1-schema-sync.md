# Phase 1: DB Schema Synchronization & Alembic Migrations

## Root Cause of 500 Errors
The application was using `Base.metadata.create_all()` in the FastAPI `lifespan` function. While this is sufficient for initial table creation, it does **not** alter existing tables. When developers previously modified the SQLAlchemy models (e.g. dropping old coordinate fields, modifying `JSONB` structures, or adding `description`), the underlying Postgres database was unaware of the changes.

The API routes (like `/export`) executing `select(Marker)` expect the database to return exactly the columns mapped by the SQLAlchemy model. Because columns were dropped from the `Marker` model, SQLAlchemy no longer selected them, but if Pydantic schemas still referenced them or if other schema drift occurred, it caused serialization crashes (500s). In addition, previous developers added raw `ALTER TABLE` statements inside `main.py` which is an unsafe anti-pattern.

## Mismatches Found & Migrated
When generating the baseline Alembic migration, the following schema drift was identified and resolved in the database:
- **`sessions.status`**: Modified to become `NOT NULL` with a default of `'active'`.
- **`markers` JSON fields**: `capture_payload`, `coordinates`, `target`, `source`, `screenshots`, and `diagnostics` were migrated from SQLAlchemy text-based `JSONB` to native `JSON` columns.
- **Removed fields on `markers`**: `client_x`, `client_y`, `viewport_height`, `viewport_width`, `width`, `height`, `review_token`, `deleted_at`, `reviewer_name`, `client_color`, `color_index`, `color_token`, `source_type`, and `reviewer_id`.

## Migration Strategy Adopted
1. Installed and initialized `alembic` in the backend.
2. Configured `alembic/env.py` to strip out the `+asyncpg` driver requirement so Alembic can use synchronous `psycopg2` under the hood for schema manipulation, reading the `DATABASE_URL` dynamically.
3. Generated a baseline autogenerated revision to sync all models.
4. Executed `alembic upgrade head` to align the database.

## Startup Logic Removed
- Removed the unsafe raw `ALTER TABLE sessions ADD COLUMN IF NOT EXISTS ...` commands from `main.py`'s lifespan boot cycle.

## Affected Routes
- `/export/session/{id}/markdown` (and CSV)
- Any route hitting `GET /projects` or `GET /sessions` that queries all Markers or Sessions.

## Known Compatibility Risks
- We dropped `client_x`, `client_y`, and related legacy fields from the DB. If any hidden legacy script queries those raw columns from Postgres, it will fail.
- Moving forward, **any change to `models/core.py` must be accompanied by an Alembic migration (`alembic revision --autogenerate`).** `create_all` will no longer auto-patch things for developers.
